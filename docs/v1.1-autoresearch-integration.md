# v1.1 — Autoresearch 패턴 통합 설계

> Karpathy의 [autoresearch](https://github.com/karpathy/autoresearch) 핵심 패턴을 분석하고, Solo Founder Agents에 적용하는 방안을 정리한다.

---

## 1. Autoresearch 핵심 구조

3개 파일로 구성된 자율 실험 시스템:

| 파일 | 역할 | 수정 주체 |
|------|------|----------|
| `prepare.py` | 데이터/평가 기반 (불변) | 없음 |
| `train.py` | 모델/학습 코드 | AI |
| `program.md` | 연구 방향/제약 조건 | 사람 |

### 작동 루프

```
AI가 train.py 수정 → 5분 학습 → 메트릭 측정 → 개선됐나? → keep/discard → 반복
```

- 밤새 ~100회 실험, 아침에 결과 확인
- 실험당 ~6분 (학습 5분 + 오버헤드 1분)

### 설계 원칙

| 원칙 | 설명 |
|------|------|
| **단일 메트릭** | `val_bpb` 하나로 개선 여부 판단. 주관 개입 없음 |
| **고정 시간 예산** | 모든 실험 5분. 공정한 비교 보장 |
| **범위 격리** | `train.py`만 수정. 평가 기반(`prepare.py`)은 불변 |
| **Git = 메모리** | 커밋이 실험 기록. 실패 시 `git reset`, 성공만 브랜치에 잔존 |
| **program.md = 조직 코드** | 자연어로 연구 방향 설정. 코드가 아닌 의도를 전달 |
| **자율 반복** | `NEVER STOP` 지시. 사람 개입 없이 무한 루프 |

### 상태 관리

```
Git 커밋 히스토리  → 코드 상태 추적 (성공 실험만 전진)
results.tsv       → 모든 실험 로그 (keep/discard/crash 전부 기록)
run.log           → 현재 실험의 stdout (매번 덮어씀)
```

`results.tsv` 형식:
```tsv
commit	val_bpb	memory_gb	status	description
a1b2c3d	0.997900	44.0	keep	baseline
d4e5f6g	0.997500	45.2	keep	improved architecture
h7i8j9k	0.998100	43.8	discard	worse performance
```

## 2. 우리 시스템과의 매핑

### 구조 대응

| Autoresearch | Solo Founder Agents | 역할 |
|-------------|---------------------|------|
| `prepare.py` | `agents/{agent}/SKILL.md` | 불변 기반. 에이전트 역량/역할 정의 |
| `train.py` | `projects/{id}/output/` | AI가 수정하는 산출물 |
| `program.md` | `projects/{id}/program.md` | 사람이 정의하는 실행 방향 |
| `results.tsv` | `projects/{id}/results.tsv` | 실험 로그 (누적, append-only) |
| Git 커밋 | Git 커밋 | 실험 상태 추적 + 롤백 |

### 현재 한계 vs Autoresearch 해결

| 현재 | 문제 | Autoresearch 방식 적용 |
|------|------|----------------------|
| 루틴이 보고만 함 | 인사이트 → 행동 연결 없음 | keep/discard 판단 + 자동 반복 |
| 에이전트가 일회성 | 매번 사람이 지시 | program.md로 자율 반복 실행 |
| 실험 추적 없음 | 뭘 시도했고 뭐가 실패했는지 모름 | results.tsv + Git monotonic ratchet |
| 주관적 판단 | AI가 "좋은 것 같습니다" 수준 | 단일 메트릭으로 기계적 판단 |

## 3. 적용 방안

### 방안 1: `program.md` 패턴 도입

현재 에이전트는 SKILL.md(역량)만 있고, 실행 방향은 매번 사람이 지시한다. program.md를 도입하면 에이전트가 **자율적으로 반복 실행** 가능.

```
agents/SKILL.md        → 에이전트의 역량 (불변, prepare.py)
projects/program.md    → 이번 프로젝트의 실행 방향 (사람이 수정, program.md)
projects/output/       → AI가 수정하는 산출물 (train.py)
```

**program.md 예시:**

```markdown
# Research Program: 전세사기 방지 앱 PMF 검증

## 목표
전세사기 불안감을 가진 2030 사회초년생의 핵심 니즈를 파악한다.

## 메트릭
- signal_relevance: 시그널의 타겟 고객 관련성 (0-1, 높을수록 좋음)
- insight_actionability: 인사이트의 실행 가능성 (0-1)

## 범위
- 수정 가능: output/research-report.md, output/persona.md
- 수정 불가: SKILL.md, TEAM_KNOWLEDGE.md

## 데이터 소스
- 부동산 커뮤니티 (직방, 피터팬의 좋은방 구하기)
- 뉴스 기사 (전세사기 관련)
- 정부 정책 발표

## 루프
1. 데이터 수집 (Desk Researcher)
2. 분석 및 인사이트 추출 (Data Analyst)
3. 메트릭 측정
4. signal_relevance > 0.7이면 keep, 아니면 discard
5. 반복
```

### 방안 2: 루틴에 keep/discard + 누적 메트릭

현재 루틴 흐름:
```
Signal Scan → 시그널 보고 → 끝
```

개선 흐름:
```
Signal Scan → 시그널 보고 → 가설 자동 생성 → 검증 방법 제안
  → 다음 Signal Scan에서 결과 평가 → 가설 상태 자동 업데이트
  → results.tsv에 기록 → 반복
```

**results.tsv 형식 (비즈니스 실험용):**

```tsv
date	hypothesis_id	metric	value	status	description
2026-03-18	h-001	signup_rate	0.032	keep	CTA 문구 "무료 체험" → "지금 시작"
2026-03-19	h-002	signup_rate	0.028	discard	가격 표시 제거 → 전환율 하락
2026-03-20	h-003	bounce_rate	0.45	keep	히어로 섹션 카피 변경
```

루틴이 **보고 시스템**에서 **학습 시스템**으로 진화.

### 방안 3: `solo-agents experiment` CLI

autoresearch 패턴을 직접 구현하는 CLI 명령:

```bash
solo-agents experiment --product my-saas --program program.md
```

실행 흐름:
```
1. program.md 읽기 (연구 방향, 메트릭, 범위)
2. Git 브랜치 생성 (experiment/{timestamp})
3. 베이스라인 실행 → 초기 메트릭 기록
4. 루프 시작:
   a. 이전 결과 분석 (results.tsv + git log)
   b. 다음 실험 설계
   c. 산출물 수정 (output/ 내 파일만)
   d. Claude Code 실행 → 메트릭 측정
   e. keep/discard 판단
   f. Git 커밋 (keep) 또는 git reset (discard)
   g. results.tsv에 기록
   h. 반복
```

**시간 예산:** autoresearch의 5분 학습 대신, Claude Code 실행 1회 = 1 실험 사이클.

### 방안 4: Git 기반 실험 추적

현재 JSONL 메모리와 별개로, Git 커밋 자체를 실험 기록으로 활용:

```bash
# 에이전트 실행이 구조화된 커밋 메시지로 기록
git log --oneline
a1b2c3d [experiment] h-003 bounce_rate=0.45 keep: 히어로 카피 변경
d4e5f6g [experiment] h-002 signup_rate=0.028 discard: 가격 표시 제거
h7i8j9k [experiment] h-001 signup_rate=0.032 keep: CTA 문구 변경
```

**Monotonic improvement ratchet:**
- 성공한 실험만 브랜치에 남김
- 실패한 실험은 revert하되 results.tsv에는 기록
- Git 히스토리 = 최적 경로의 기록

### 방안 5: 하룻밤 자율 리서치 모드

방안 1-4를 통합한 최종 형태:

```bash
solo-agents research --product my-saas --program program.md --hours 8
```

```
[22:00] 리서치 시작
  → Desk Researcher: 시장 데이터 수집 (실험 1-5)
  → Data Analyst: 데이터 분석 (실험 6-10)
  → User Researcher: 페르소나 정제 (실험 11-15)
  → ...반복...
[06:00] 리서치 완료
  → results.tsv (50회 실험 기록)
  → 분석 보고서 자동 생성
  → 메신저에 요약 전송
```

아침에 일어나면:
- 밤새 수행한 실험 목록 + keep/discard 결과
- 가장 효과적이었던 방향 하이라이트
- 다음 단계 제안

## 4. 구현 우선순위

| 순위 | 방안 | 난이도 | 변경 범위 | 효과 |
|------|------|--------|----------|------|
| **1** | program.md 패턴 | 낮음 | 템플릿 추가 + 문서화 | 에이전트 자율성 대폭 향상 |
| **2** | Git 기반 실험 추적 | 낮음 | 커밋 메시지 규칙 추가 | 이미 Git 사용 중, 비용 거의 없음 |
| **3** | 루틴 keep/discard 메트릭 | 중간 | scheduler 로직 수정 | 루틴이 보고→학습으로 진화 |
| **4** | `solo-agents experiment` | 중간 | 새 CLI 명령 + 실험 엔진 | autoresearch 패턴 직접 적용 |
| **5** | 하룻밤 자율 리서치 | 높음 | 방안 1-4 통합 + 에이전트 체이닝 | 최종 목표 |

### Phase 1 구현 (방안 1+2)

변경 파일:
- `assets/templates/program.md` — program.md 템플릿 추가
- `assets/templates/results.tsv` — results.tsv 헤더 템플릿
- `docs/architecture.md` — program.md 패턴 문서화

### Phase 2 구현 (방안 3+4)

변경 파일:
- `src/cli/experiment.ts` — 새 CLI 명령
- `src/scheduler/routines.ts` — keep/discard 로직 추가
- `src/scheduler/memory.ts` — results.tsv 지원

### Phase 3 구현 (방안 5)

변경 파일:
- `src/cli/research.ts` — 자율 리서치 모드
- `src/bot/agent-chainer.ts` — 에이전트 체이닝 엔진
