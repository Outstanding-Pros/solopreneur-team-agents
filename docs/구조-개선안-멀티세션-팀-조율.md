# 구조 개선안: Claude Code 멀티 세션 팀 조율

> 현재 단일 세션 기반 순차 실행 구조의 한계를 분석하고, 멀티 세션 병렬 조율 구조로의 개선안을 제시합니다.

---

## 1. 현재 구조의 한계

### 1.1 단일 세션 병목

현재 Orchestrator가 모든 에이전트를 하나의 Claude Code 세션에서 순차 호출합니다.

```
[하나의 세션]
Orchestrator → Agent A → Agent B → Agent C → ...
                 ↓          ↓          ↓
              output.md  output.md  output.md
```

**문제점:**
- 컨텍스트 윈도우를 모든 에이전트가 공유 → 후반부 에이전트는 초반 맥락 유실
- 순차 실행만 가능 → 병렬 가능한 작업도 직렬화
- 한 에이전트 실행 중 다른 에이전트 산출물 참조/수정 불가
- 세션 중단 시 전체 워크플로우 재시작 필요

### 1.2 에이전트 간 핸드오프 미정의

각 에이전트 SKILL.md에 "다음 에이전트"와 "산출물"이 정의되어 있지만, 실제로 세션 간 맥락 전달을 위한 구조화된 프로토콜이 없습니다.

- 산출물 파일은 있지만, 어떤 맥락에서 생성되었는지 메타 정보 부재
- 에이전트 A의 결정 근거를 에이전트 B가 파악할 방법 없음
- 수정 요청 시 영향 범위 추적 어려움

### 1.3 Orchestrator와 실제 에이전트 경로 불일치

`orchestrator/SKILL.md`의 Integration 섹션이 실제 디렉토리 구조와 다릅니다:

```
# SKILL.md에 정의된 경로 (존재하지 않음)
agents/research/SKILL.md
agents/branding/SKILL.md
agents/planning/SKILL.md

# 실제 구조 (팀 기반)
agents/experience/user-researcher/SKILL.md
agents/growth/brand-marketer/SKILL.md
agents/strategy/feature-planner/SKILL.md
```

### 1.4 팀 간 병렬 작업 미활용

현재 워크플로우는 완전 직렬입니다. 하지만 실제로는 병렬 가능한 작업 쌍이 존재합니다:

| 워크플로우 단계 | 병렬 가능한 작업 |
|---------------|----------------|
| Research 단계 | User Researcher + Desk Researcher |
| Planning 단계 | Feature Planner + Scope Estimator |
| Design 단계 | UX Designer가 완료 후 UI Designer + Content Writer 동시 |
| Development 단계 | Frontend + Backend 동시 개발 |
| Launch 단계 | GTM Strategist + Paid Marketer + Content Writer |

## 2. 개선안: 멀티 세션 팀 조율 아키텍처

### 2.1 핵심 개념

```
┌──────────────────────────────────────────────────┐
│              Orchestrator 세션 (Main)              │
│  - PRD 관리                                       │
│  - 워크플로우 상태 관리                              │
│  - 세션 간 조율                                    │
└────────┬─────────────┬──────────────┬─────────────┘
         │             │              │
    ┌────▼────┐  ┌────▼─────┐  ┌────▼─────┐
    │ 세션 A  │  │  세션 B   │  │  세션 C   │
    │(팀/에이전트)│  │(팀/에이전트) │  │(팀/에이전트) │
    └─────────┘  └──────────┘  └──────────┘
         │             │              │
         └─────────────┼──────────────┘
                       ▼
              projects/{id}/ (공유 파일시스템)
```

**원칙:**
1. Orchestrator 세션은 조율만 담당 (실행 X)
2. 각 에이전트/팀은 독립 세션에서 실행
3. 파일 시스템이 세션 간 통신 채널
4. 핸드오프 파일이 맥락 전달 표준

### 2.2 세션 분리 전략

#### Option A: 팀 단위 세션 (권장)

```
세션 1: Orchestrator       → PRD 생성, 워크플로우 관리
세션 2: Strategy Team      → PMF Planner, Feature Planner, ...
세션 3: Experience Team    → User Researcher, UX Designer, ...
세션 4: Growth Team        → GTM Strategist, Content Writer, ...
세션 5: Engineering Team   → Architect, FDE, Frontend, Backend, ...
```

- 팀 내 에이전트는 TEAM_KNOWLEDGE.md를 공유하므로 하나의 세션에서 자연스러움
- 세션 수가 관리 가능 (최대 5개)
- 팀 내 순차 실행, 팀 간 병렬 실행

#### Option B: 에이전트 단위 세션

```
각 에이전트마다 독립 세션
→ 최대 23개 세션
```

- 최대 병렬성이지만 조율 복잡도 급증
- 특수한 경우(대규모 프로젝트)에만 권장

#### Option C: 단계(Phase) 단위 세션

```
Phase별로 관련 에이전트를 묶어서 실행
→ Research Phase: User Researcher + Desk Researcher (1세션)
→ Planning Phase: Feature Planner + Scope Estimator (1세션)
→ Design Phase: UX Designer + UI Designer (1세션)
→ ...
```

- 워크플로우 단계에 맞춰서 세션 생성/종료
- 컨텍스트 윈도우 효율적 사용

### 2.3 세션 간 통신 프로토콜

#### 핸드오프 파일 표준 (`_handoff.md`)

각 에이전트가 작업 완료 시 산출물과 함께 핸드오프 파일을 생성합니다.

```markdown
# Handoff: [출발 에이전트] → [도착 에이전트]

## Meta
- from: user-researcher
- to: ux-designer
- project: {project_id}
- stage: stage-1-research
- created: 2026-02-25T10:00:00+09:00
- version: v1

## Summary
핵심 발견 사항 3줄 요약

## Artifacts
- `stage-1-research/research-report.md` - 리서치 리포트
- `stage-1-research/persona.md` - 페르소나 정의
- `stage-1-research/journey-map.md` - 사용자 여정 맵

## Key Decisions
| 결정 사항 | 근거 | 대안 |
|----------|------|------|
| 초기 타겟: 사회초년생 | 인터뷰 5건 중 4건 일치 | 신혼부부 (2순위) |

## Context for Next Agent
- 페르소나 "지은"을 기준으로 플로우 설계 요청
- 모바일 퍼스트 (인터뷰 대상 90% 모바일 사용)
- 정보 입력 최소화가 핵심 (3개 이하 필드)

## Open Questions
- [ ] 공인중개사 제휴 모델은 Strategy팀 확인 필요
- [ ] 데이터 출처(등기부등본 API) 기술 검토 필요

## Dependencies
- `strategy/feature-planner`의 MVP 범위 확정 후 진행
```

#### 워크플로우 상태 파일 (`_status.yaml`)

프로젝트 루트에 전체 워크플로우 상태를 추적하는 파일을 둡니다.

```yaml
# projects/{project_id}/_status.yaml

project:
  id: jeonse-guard
  type: pmf
  created: 2026-02-25
  version: v1

workflow:
  - stage: research
    agents: [user-researcher, desk-researcher]
    status: completed  # pending | in_progress | completed | needs_revision
    session: session-2  # 어느 세션에서 실행했는지
    started: 2026-02-25T09:00:00
    completed: 2026-02-25T10:30:00
    artifacts:
      - stage-1-research/research-report.md
      - stage-1-research/persona.md
    handoff: stage-1-research/_handoff.md

  - stage: planning
    agents: [feature-planner, scope-estimator]
    status: in_progress
    session: session-3
    started: 2026-02-25T10:35:00
    blocked_by: []

  - stage: design
    agents: [ux-designer, ui-designer]
    status: pending
    blocked_by: [planning]

  - stage: development
    agents: [fde]
    status: pending
    blocked_by: [design]

  - stage: marketing
    agents: [gtm-strategist, content-writer]
    status: pending
    blocked_by: [planning]  # design 완료 전에 시작 가능
```

### 2.4 세션별 CLAUDE.md 자동 생성

각 세션이 시작될 때, 해당 팀/에이전트에 맞는 컨텍스트를 자동으로 로드하기 위해 세션별 CLAUDE.md를 생성합니다.

```
projects/{project_id}/
├── _status.yaml
├── prd.md
├── sessions/
│   ├── orchestrator/CLAUDE.md     # Orchestrator 세션용
│   ├── strategy/CLAUDE.md         # Strategy 팀 세션용
│   ├── experience/CLAUDE.md       # Experience 팀 세션용
│   ├── growth/CLAUDE.md           # Growth 팀 세션용
│   └── engineering/CLAUDE.md      # Engineering 팀 세션용
```

**예시: `sessions/experience/CLAUDE.md`**
```markdown
# Experience Team Session

## Project Context
- PRD: ../../prd.md 참조
- 현재 단계: Design (Stage 3)
- 이전 단계 산출물: ../../stage-1-research/_handoff.md 참조

## Your Agents
1. UX Designer: agents/experience/ux-designer/SKILL.md
2. UI Designer: agents/experience/ui-designer/SKILL.md

## Team Knowledge
agents/_teams/experience/TEAM_KNOWLEDGE.md 참조

## Current Task
핸드오프 파일의 "Context for Next Agent" 섹션에 따라:
- 페르소나 "지은" 기준 사용자 플로우 설계
- 모바일 퍼스트 와이어프레임 작성
- 완료 시 _handoff.md 작성하여 Engineering 팀에 전달

## Output Rules
- 모든 산출물은 ../../stage-3-design/ 아래에 저장
- 완료 시 _handoff.md 작성 필수
- ../../_status.yaml의 해당 stage를 completed로 업데이트
```

## 3. 개선된 워크플로우

### 3.1 프로젝트 시작 (Orchestrator 세션)

```
Step 1: 사용자 아이디어 입력
Step 2: 구체화 질문 (Phase 1)
Step 3: PRD 생성 (Phase 2)
Step 4: _status.yaml 생성
Step 5: 세션별 CLAUDE.md 생성
Step 6: 사용자에게 다음 세션 실행 안내
```

### 3.2 병렬 실행 패턴

#### PMF 탐색 워크플로우 (개선 후)

```
Phase 1 - 리서치 (병렬)
├── [Experience 세션] User Researcher + Desk Researcher
└── [Strategy 세션] Idea Refiner (동시 가능)

Phase 2 - 기획 + 브랜딩 (병렬)
├── [Strategy 세션] PMF Planner → Feature Planner
└── [Growth 세션] Brand Marketer (브랜드 중요 시)

Phase 3 - 디자인
└── [Experience 세션] UX Designer → UI Designer

Phase 4 - 개발 + 마케팅 (병렬)
├── [Engineering 세션] FDE or Frontend + Backend
└── [Growth 세션] GTM Strategist + Content Writer

Phase 5 - 런칭
└── [Growth 세션] Paid Marketer
```

**Before (직렬):**
```
A → B → C → D → E → F → G → H  (8 단계 순차)
총 시간: 8T
```

**After (병렬):**
```
Phase 1: [A, B] 동시    → 1T
Phase 2: [C, D] 동시    → 1T
Phase 3: [E]            → 1T
Phase 4: [F, G] 동시    → 1T
Phase 5: [H]            → 1T
총 시간: 5T (37.5% 단축)
```

### 3.3 수정 요청 처리 (개선)

```
사용자: "리서치 결과 수정해줘"

Orchestrator:
1. _status.yaml에서 영향 받는 stage 식별
2. 해당 stage 이후를 needs_revision으로 변경
3. 영향 받는 세션에게 _handoff.md 업데이트로 알림
4. 수정 완료 후 의존 stage만 선택적 재실행
```

### 5.2 루트 CLAUDE.md 생성

프로젝트 루트에 CLAUDE.md를 추가하여, 어떤 세션이든 프로젝트 구조와 규칙을 자동 인식하도록 합니다.

```markdown
# Solo Founder Agents

## 프로젝트 구조
- orchestrator/SKILL.md: 프로젝트 시작점

4. 산출물 생성 후 _handoff.md 작성
5. _status.yaml 업데이트
```

## 6. 실행 가이드: 멀티 세션 운영법

### 6.1 터미널 레이아웃

```
┌────────────────────┬────────────────────┐
│                    │                    │
│  Orchestrator      │  Team A Session    │
│  (조율/모니터링)     │  (작업 실행)         │
│                    │                    │
├────────────────────┼────────────────────┤
│                    │                    │
│  Team B Session    │  Team C Session    │
│  (작업 실행)         │  (작업 실행)         │
│                    │                    │
└────────────────────┴────────────────────┘
```

### 6.2 세션 시작 순서

```bash
# 1. Orchestrator 세션 (PRD 생성)
cd solopreneur-agents
claude

# 2. PRD 승인 후, 팀 세션들 시작
# 터미널 2: Experience Team
cd solopreneur-agents
claude --prompt "projects/{id}/sessions/experience/CLAUDE.md 읽고 작업 시작"

# 터미널 3: Strategy Team (Research와 병렬)
cd solopreneur-agents
claude --prompt "projects/{id}/sessions/strategy/CLAUDE.md 읽고 작업 시작"

# 터미널 4: Growth Team (브랜드 중요 시)
cd solopreneur-agents
claude --prompt "projects/{id}/sessions/growth/CLAUDE.md 읽고 작업 시작"
```

### 6.3 진행 상황 확인 (Orchestrator에서)
