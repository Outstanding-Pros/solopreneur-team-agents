# v1.2 — 웹 대시보드 및 커뮤니티 확장 설계

> 프로젝트 모니터링 대시보드를 웹으로 구축하고, 커뮤니티 → 코워킹 플랫폼으로 확장하는 전략을 정리한다.

---

## 1. 현재 상태와 확장 동기

### 현재 모니터링 수단

| 수단 | 한계 |
|------|------|
| `solo-agents status` CLI | 터미널 전용. 시각화 없음. 로컬에서만 확인 |
| 메신저 채널 보고 | 텍스트 기반. 전체 현황 조망 어려움. 히스토리 검색 불편 |
| JSONL 메모리 파일 | 직접 열어봐야 함. 트렌드/패턴 파악 불가 |

### 확장 비전

```
Phase 1: 웹 대시보드 (모니터링)
Phase 2: API 서버 (CLI ↔ 웹 동기화)
Phase 3: 커뮤니티 (제품 홍보, 피드백)
Phase 4: 코워킹 공간 (메타버스/실시간 협업)
```

---

## 2. 리포지토리 분리 전략

### 왜 같은 리포에 넣으면 안 되는가

| 문제 | 설명 |
|------|------|
| **배포 단위가 다름** | CLI는 `npm install -g`, 웹은 Vercel/AWS 호스팅 |
| **사용자가 다름** | CLI = 솔로 파운더 로컬 실행, 웹 = 불특정 다수 접속 |
| **릴리즈 주기가 다름** | CLI 기능 추가와 웹 UI 수정은 독립적 |
| **의존성 오염** | React/Next.js + DB + 인증이 CLI npm 패키지에 포함됨 |
| **npm 패키지 사이즈** | 웹 코드가 `npm install -g`에 딸려오면 불필요하게 무거워짐 |

커뮤니티/메타버스까지 확장하면 **SaaS 플랫폼**이 되며, 이는 CLI 도구와 완전히 다른 프로덕트다.

### 리포 구조

```
Organization: Outstanding-Pros/
├── solopreneur-team-agents     ← 현재 리포 (CLI npm 패키지)
├── solopreneur-dashboard       ← 웹 대시보드 (새 리포)
└── solopreneur-api             ← 공유 API 서버 (필요 시)
```

| 리포 | 역할 | 스택 | 배포 |
|------|------|------|------|
| **team-agents** | CLI 도구, 봇, 스케줄러 | TypeScript, npm | `npm install -g` |
| **dashboard** | 웹 대시보드 → 커뮤니티 → 코워킹 | Next.js, DB, 인증 | Vercel / AWS |
| **api** | CLI ↔ 웹 데이터 동기화 | API 서버 | Cloud |

## 3. 연결 아키텍처

### CLI ↔ 웹 데이터 흐름

```
[CLI: solo-agents]                    [Web: dashboard]
  봇 실행, 루틴, 실험                     대시보드, 커뮤니티, 코워킹
       │                                      │
       └──────── API (sync) ──────────────────┘
                 - 프로젝트 상태
                 - 실험 결과 (results.tsv)
                 - 루틴 로그
                 - 메모리 (JSONL)
```

### 동기화 방식

```
옵션 A: Push 방식 (CLI → API)
  - 루틴 완료 시 자동 push
  - solo-agents sync 수동 명령
  - 장점: 실시간. 단점: API 서버 필요

옵션 B: Pull 방식 (웹 ← Git)
  - 웹이 Git 리포에서 직접 읽기
  - JSONL/results.tsv 파싱
  - 장점: API 불필요. 단점: Git push 필요

옵션 C: 하이브리드
  - Phase 1은 Pull (Git 기반, 빠른 구현)
  - Phase 2부터 Push (API 추가, 실시간)
```

**권장: 옵션 C (하이브리드)**
- 초기에 API 서버 없이 빠르게 시작
- 사용자 늘면 API로 전환

### 독립 사용 보장

| 조합 | 가능 여부 |
|------|----------|
| CLI만 (웹 없이) | 가능. 현재와 동일 |
| 웹만 (CLI 없이) | 가능. 웹에서 직접 데이터 입력 |
| CLI + 웹 | 가능. API로 동기화 |

## 4. Phase별 상세 설계

### Phase 1: 웹 대시보드 (모니터링)

**목적:** `solo-agents status`의 웹 버전. 프로젝트/루틴/실험 현황 시각화.

**핵심 페이지:**

| 페이지 | 내용 |
|--------|------|
| 홈 | 전체 제품 목록 + 요약 카드 |
| 제품 상세 | 프로젝트 목록, 활성 가설, 최근 결정 |
| 루틴 로그 | 루틴별 실행 이력 + 트렌드 차트 |
| 실험 추적 | results.tsv 시각화, keep/discard 비율, 메트릭 추이 |
| 메모리 뷰어 | signals/experiments/decisions JSONL 검색 + 필터 |

**스택:**
```
Next.js (App Router)
├── Tailwind CSS + shadcn/ui
├── Recharts (차트)
├── 데이터: Git 리포 직접 읽기 또는 JSON 파일
└── 인증: 없음 (Phase 1은 로컬/개인용)
```

### Phase 2: API 서버 (데이터 동기화)

**목적:** CLI와 웹 간 실시간 데이터 동기화.

**API 엔드포인트:**

```
POST /api/sync/routine-log    ← CLI가 루틴 완료 시 push
POST /api/sync/experiment     ← 실험 결과 push
GET  /api/products            → 제품 목록
GET  /api/products/:slug/memory → 메모리 데이터
GET  /api/products/:slug/experiments → 실험 이력
```

**CLI 측 변경:**
```bash
# .env에 추가
DASHBOARD_API_URL=https://my-dashboard.vercel.app/api
DASHBOARD_API_KEY=...
```

스케줄러/루틴 완료 시 자동으로 API에 push.

**스택:**
```
Next.js API Routes 또는 별도 서버
├── PostgreSQL / Supabase
├── API Key 인증
└── WebSocket (실시간 업데이트)
```

### Phase 3: 커뮤니티 (제품 홍보, 피드백)

**목적:** 솔로 파운더들이 자기 제품을 공유하고 피드백을 주고받는 공간.

**핵심 기능:**

| 기능 | 설명 |
|------|------|
| 제품 쇼케이스 | 제품 페이지 (brief, 스크린샷, 링크) |
| 피드백 요청 | "이 가설 어떻게 생각하세요?" 스레드 |
| 실험 공유 | results.tsv 공개 + 다른 파운더의 코멘트 |
| 에이전트 스킬 공유 | 커스텀 SKILL.md 마켓플레이스 |
| 위클리 쇼앤텔 | 자동 생성된 Weekly Review 공유 |

**추가 스택:**
```
├── 사용자 인증 (OAuth: Google, GitHub)
├── 프로필 / 팔로우
├── 댓글 / 리액션
└── 알림
```

### Phase 4: 코워킹 공간 (메타버스/실시간 협업)

**목적:** 솔로 파운더들이 가상 공간에서 함께 일하는 느낌을 받을 수 있는 환경.

**핵심 기능:**

| 기능 | 설명 |
|------|------|
| 가상 오피스 | 2D/3D 공간에서 아바타로 존재 |
| 실시간 상태 | "지금 실험 진행 중", "리서치 중" 표시 |
| 포커스 세션 | 함께 집중하는 뽀모도로 타이머 |
| 실시간 채팅/음성 | 공간 기반 proximity chat |
| AI 팀 시각화 | 내 AI 에이전트들이 일하는 모습 시각화 |

**추가 스택:**
```
├── WebSocket / WebRTC
├── Canvas 또는 Three.js (공간 렌더링)
├── 실시간 상태 동기화
└── 음성/영상 (선택)
```

## 5. 데이터 모델 (Phase 1-2 기준)

### 핵심 엔티티

```
User
├── id, name, email, avatar
└── products: Product[]

Product
├── id, slug, name, brief
├── memory: Memory
├── projects: Project[]
└── routineLogs: RoutineLog[]

Project
├── id, name, type, status
├── experiments: Experiment[]
└── handoffs: Handoff[]

Experiment (results.tsv 행)
├── id, date, hypothesis_id
├── metric, value, status (keep/discard/crash)
└── description

RoutineLog
├── id, routineId, date
├── result (markdown)
└── extractedSignals: Signal[]

Memory
├── hypotheses.jsonl
├── experiments.jsonl
├── decisions.jsonl
└── signals.jsonl
```

## 6. 구현 우선순위

| 순위 | 작업 | 리포 | 의존성 |
|------|------|------|--------|
| 1 | dashboard 리포 생성 + Next.js 초기화 | dashboard | 없음 |

| 6 | 인증 + 커뮤니티 기능 (Phase 3) | dashboard | API + DB |
| 7 | 코워킹 공간 (Phase 4) | dashboard | Phase 3 완료 |
